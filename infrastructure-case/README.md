# Развертывание распределенной веб-инфраструктуры (Flask + PostgreSQL + Prometheus/Grafana)

## Цель проекта
Перенести веб-приложение на Flask из локальной среды в распределенную инфраструктуру, приближенную к боевой. Моей задачей было не просто запустить код, а настроить безопасное взаимодействие между узлами, автоматизировать работу служб и обеспечить полный мониторинг состояния системы. Все работы проводились в рамках домашней лаборатории на базе виртуальных машин с Ubuntu Server.

## Стек технологий
* **ОС:** Linux (Ubuntu Server)
* **Web/App:** Nginx (Reverse Proxy), Gunicorn, Flask, Python 3.12
* **База данных:** PostgreSQL 16, SQLAlchemy, Alembic
* **Мониторинг:** Prometheus, Grafana, Node Exporter
* **Администрирование:** Systemd, UFW, Bash, SQL



## Архитектура и описание серверов
Инфраструктура разделена на три независимых узла для повышения отказоустойчивости и безопасности:

* **App Server (192.168.56.10) — Точка входа и логика.** Nginx принимает внешние HTTP-запросы на 80 порту и проксирует их на Gunicorn (порт 8000), который управляет Flask-приложением. Для поддержания работы приложения в фоне написан кастомный `systemd` юнит.
* **DB Server (192.168.56.12) — Изолированная база данных.**
  Выделенный сервер с PostgreSQL. Настроен файл `pg_hba.conf` таким образом, чтобы база принимала подключения исключительно от сервера приложений (по порту 5432), отсекая любые сторонние запросы.
* **Monitoring Server (192.168.56.13) — Мониторинг.**
  Выделенный сервер для мониторинга. Prometheus собирает метрики через агентов (Node Exporter) с серверов App и DB, а Grafana визуализирует эти данные (нагрузка на CPU, память, диск, сеть) на наглядных дашбордах.

## С какими проблемами столкнулся и как решал (Troubleshooting)
В процессе развертывания я столкнулся с рядом инфраструктурных и программных проблем, которые успешно диагностировал и решил:

* **Миграция легаси-кода на Python 3.12:**
  При обновлении стека сломались миграции базы данных Alembic. Выяснил, что проблема кроется в устаревших типах данных библиотеки SQLAlchemy. Вручную провел рефакторинг моделей.
* **Права доступа в PostgreSQL 16:**
  Приложение не могло записать данные, выдавая ошибку `InsufficientPrivilege`. Изучил документацию и выяснил, что в новых версиях Postgres изменилась политика безопасности по умолчанию. Выдал необходимые права роли приложения на схему `public` через консоль `psql`.
* **Асинхронная отправка писем и SSL-конфликты:**
  Фоновые потоки приложения падали с ошибками `SMTPServerDisconnected` и `ssl.SSLEOFError`. Провел диагностику SMTP-соединения, разобрался в тонкостях работы TLS/SSL на 465 и 587 портах. Чтобы обойти блокировку входа из-за неподтвержденной почты на этапе отладки, подключился к БД напрямую и SQL-запросом (`UPDATE users SET confirmed = true`) вручную активировал тестовых пользователей.
* **Сетевая отладка и автоматизация:**
  Для диагностики проблем со связью между виртуалками (ошибки `No route to host`) применял сетевые утилиты `nc` (netcat), `ping` и `ss`. Настроил правила файрвола `UFW` и перевел ручной запуск Gunicorn в системную фоновую службу (`systemd`) с правилом автоматического перезапуска (`Restart=always`).
